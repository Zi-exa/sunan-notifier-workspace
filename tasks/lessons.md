# Lessons Learned

- Saat user memberi koreksi "fitur X kurang", perlakukan sebagai gap fungsional prioritas dan selesaikan end-to-end (entry point UI, layar utama terkait, notifikasi/otomasi jika relevan, lalu re-verify typecheck+lint).
- Saat user bilang mode tidak sesuai `.env`, validasi nilai file env yang benar-benar dibaca runtime (terminal) selain editor buffer, lalu tampilkan indikator runtime di UI agar verifikasi lapangan cepat.
- Untuk expo-notifications di Expo Go (SDK 53+), selalu skip registrasi remote push token berdasarkan execution environment dan tangani error unsupported secara graceful.
- Untuk integrasi Moodle, jangan asumsikan semua evaluasi ada di `mod_assign_get_assignments`; activity `quiz` juga wajib diambil agar UTS/UAS berbasis quiz muncul di task list.
- Untuk status quiz Moodle, jangan infer dari deadline saja; baca `mod_quiz_get_user_attempts` dan aktifkan refetch saat app kembali fokus supaya status selesai cepat tersinkron setelah user selesai quiz di browser.
- Untuk warning Expo Go SDK53, guard runtime saja tidak cukup jika `expo-notifications` di-import statis; gunakan lazy-load + `LogBox.ignoreLogs` sebelum module di-load agar warning remote push tidak memunculkan red log saat dev testing.
- Untuk fitur absensi, jangan hanya mengandalkan keyword judul/deskripsi; deteksi juga lewat `modulename`, `eventtype`, dan pola URL attendance karena beberapa event "daftar hadir" memakai judul generik.
- Endpoint `core_calendar_get_action_events_by_courses` dapat mengembalikan struktur `groupedbycourse`, bukan `events` flat; parser harus fleksibel dan sebaiknya digabung dengan `core_calendar_get_calendar_events` untuk menghindari event absensi hilang.
- Endpoint `core_calendar_get_calendar_events` bisa mengirim `events` sebagai object keyed-by-eventid, bukan array sequential; parser client harus menerima kedua bentuk itu agar event Upcoming Events tidak hilang.
- Saat user menegaskan fitur Absensi harus meniru `Upcoming events` SUNAN, jangan tampilkan semua modul attendance sebagai fallback; batasi hanya sesi yang punya hari/jam valid agar daftar tetap relevan.
- Saat tasklist/status dokumen bentrok dengan bukti lapangan, tetap prioritaskan bukti lapangan, tetapi pilih solusi yang masih sejalur dengan tasklist utama; untuk absensi berarti tetap berbasis data kalender/upcoming Moodle, bukan sumber fallback di luar jalur itu.
- Saat sudah punya parser robust (e.g. `toCalendarEventArray`) yang menangani variasi payload Moodle, WAJIB terapkan di SEMUA jalur yang memakai data sejenis — jangan hanya di satu jalur. Bug absensi 22-Apr-2026 terjadi karena parser hanya diterapkan di `extractActionEventsFromPayload` tapi TIDAK di `getCalendarEvents` yang langsung pakai `calendarPayload.events ?? []`.
- Modul attendance Moodle punya BANYAK sesi yang berbagi URL yang sama (e.g. `view.php?id=652636`). Jika pakai URL sebagai identity key saat merge, sesi terakhir menimpa sesi sebelumnya. HARUS pakai eventId (unik per calendar event) sebagai primary key, bukan URL modul.
- Di SUNAN UMK, `mod_attendance_get_sessions` dan `core_calendar_get_calendar_events` TIDAK tersedia/kosong. Satu-satunya sumber data absensi yang reliable adalah `core_calendar_get_calendar_upcoming_view`. Jangan asumsikan semua endpoint Moodle aktif — selalu validate dengan data lapangan.
- Saat user minta semua halaman siap sejak awal, jangan taruh preload hanya di route entry yang bisa terlewati oleh login/deep link. Tempatkan preload di area authenticated global dekat `Tabs`, lalu ikuti preferensi UX user: bila user ingin loading screen awal, pakai gate blocking; bila user ingin langsung masuk screen pertama, pakai preload background non-blocking.
- Kalau sudah ada gate preload yang menghangatkan cache, jangan tambah `refetchOnMount: 'always'` atau `refetchOnWindowFocus: 'always'` pada query yang sama. Itu hanya memunculkan spinner kedua setelah dashboard terbuka dan membuat app terasa lambat tanpa nilai UX yang sepadan.
- Untuk Expo Go dan koneksi kampus yang lambat, jangan masukkan hydration per-item yang mahal (mis. `hydrateAssignmentsWithSubmissionStatus`) ke jalur blocking screen awal. Tampilkan dulu data daftar ringan, lalu update cache di background saat status detail selesai.
- Jika QR Expo muncul tetapi Expo Go hanya menampilkan “something went wrong”, jangan langsung asumsi source app rusak. Cek dulu apakah dev server berjalan di mode `lan`; pada banyak jaringan kampus/hotspot, Expo Go perlu `expo start --tunnel` atau cache clear agar bisa menjangkau Metro.
- Jika Expo Go menampilkan `Failed to download remote update`, artinya QR berhasil dibaca tetapi bundle tidak bisa diunduh. Perlakukan itu sebagai masalah koneksi Metro/tunnel/firewall terlebih dulu, bukan bug runtime aplikasi.
- Saat user minta ganti logo aplikasi, jangan arahkan wordmark PNG lebar langsung ke semua slot icon. Turunkan minimal dua asset: `mark` persegi untuk launcher/splash dan wordmark/teks untuk screen React bila perlu.
- Saat user menyinggung mode gelap/terang pada splash atau loading screen, audit dua jalur sekaligus: config native di `app.json` (`userInterfaceStyle` + `splash.dark`) dan screen React setelah boot. Mengubah salah satu saja sering menghasilkan transisi tema yang tidak konsisten.
- Saat user mengirim screenshot awal dengan bar `Bundling xx%`, identifikasi dulu itu sebagai overlay Expo Go, bukan splash React app. Jangan janji menghapusnya dari source project; yang bisa dioptimalkan hanya screen setelah bundle selesai, atau pindah ke dev build/APK agar tidak lewat overlay Expo Go.
- Saat user bilang APK masih menampilkan splash lama, cek dulu apakah repo memakai native folder `android/`. Jika folder itu tidak ada, jangan buru-buru menyalahkan resource stale lokal; kemungkinan lebih besar build berasal dari snapshot lama atau yang terlihat adalah splash sistem Android/OEM.
- Untuk internal APK testing via EAS, hindari memakai package production yang sama jika user sering install-uninstall build dari sumber berbeda. Beri `applicationId/package` khusus untuk profile `preview/development` agar bentrok signature/package tidak memblokir instalasi di device.
- Saat user sudah mengarahkan branding visual baru, cek juga screen login/auth. Jangan berhenti di icon dan splash jika halaman pertama yang dilihat user masih terasa tidak selaras.
- Saat user memberi mockup UI yang jelas, ubah langsung komposisi utama layar agar mendekati struktur mockup (hierarki, alignment, density), bukan hanya mengganti warna atau menempelkan logo ke layout lama.
- Setelah app punya `ThemeContext`, jangan biarkan screen entry seperti login tetap memakai hardcoded palette. Screen pertama yang dilihat user harus ikut mode terang/gelap yang aktif, bukan hanya tab setelah login.
- Untuk dark mode pada screen auth, jangan hanya invert warna. Pertegas hierarki dengan panel hero, kontras input/card yang lebih dalam, dan accent yang lebih terang agar layar tidak terasa datar.
- Saat user minta area logo dibuat lebih clean, jangan pertahankan panel dekoratif hanya demi efek visual. Prioritaskan kebersihan komposisi brand block jika itu yang memang diminta user.
- Untuk file/foto yang berasal dari Moodle webservice, jangan pakai URL `pluginfile.php` mentah di client. Untuk klien token-based, normalisasikan ke `/webservice/pluginfile.php` dan sertakan `token`, atau asset seperti avatar bisa gagal dimuat walau datanya ada.
- Saat user minta halaman pengaturan dibuat lebih simple, jangan hanya mengecilkan padding atau mengganti warna. Rapikan juga hierarchy informasi: gabungkan pengaturan yang satu konteks, kurangi card yang tidak perlu, dan buat opsi utama bisa dipindai dalam sekali scroll.
- Untuk filter global yang memengaruhi query scope seperti `monitoredCourseIds`, jangan update store global pada setiap tap di screen Settings. Pakai draft lokal dan commit saat user menekan `Simpan`, agar screen tidak terasa reload sendiri saat user baru memilih opsi.
- Saat user minta screen dibuat lebih clean, hapus judul yang redundant jika konteks halaman sudah jelas, lalu pakai ikon seperlunya untuk memperjelas section/aksi. Jangan mengganti teks berlebih dengan teks lain yang sama-sama tidak perlu.
- Saat user minta ikon diterapkan di banyak halaman, mulai dari komponen shared seperti `SectionCard`, `EmptyState`, card, dan header title. Itu jauh lebih konsisten daripada menempel ikon satu per satu langsung di setiap screen.
- Saat user minta urutan berdasarkan deadline terdekat, jangan ubah urutan di satu screen saja. Perbaiki util sorting global yang dipakai query/card/dashboard, dan pakai waktu yang paling relevan untuk tiap domain: `dueDate` untuk tugas, `closesAt` atau `startsAt` untuk absensi.
- Untuk aksi penting seperti `Simpan Pengaturan`, jangan andalkan banner pasif saja. Beri feedback sukses yang eksplisit dan langsung terlihat, misalnya alert native atau toast, setelah proses benar-benar berhasil.
- Jika sebuah screen punya tombol `Simpan`, jangan campur sebagian field yang langsung menulis ke store dengan sebagian field yang memakai draft. Pakai draft lokal secara konsisten, lalu commit sekali saat save; kalau tidak ada perubahan, tampilkan alert yang jelas.
- Jika feedback penting perlu konsisten dengan desain aplikasi dan mode terang/gelap, jangan pakai `Alert.alert` native. Gunakan dialog/toast in-app yang membaca `ThemeContext`, terutama untuk alur save atau konfirmasi utama.
- Jika halaman settings punya CTA utama seperti `Simpan Pengaturan`, jangan biarkan section opsional yang panjang selalu terbuka. Lipat section seperti notifikasi atau daftar item besar, lalu tampilkan ringkasan status di header agar CTA tetap cepat terlihat.
- Saat merapikan urutan halaman settings, letakkan pengaturan perilaku aplikasi yang paling global seperti sinkronisasi lebih atas daripada section opsional detail seperti notifikasi, agar alurnya lebih cepat dipindai.
- Jika beberapa section settings memakai pola collapsible yang sama, jangan kelola dengan banyak boolean terpisah. Pakai satu state accordion seperti `expandedSection` agar perilaku satu-terbuka lebih sederhana dan konsisten saat kebutuhan bertambah.
- Jika user sudah meminta section settings dibuat collapsible/accordion, jangan berhenti di toggle state saja. Tambahkan motion ringan seperti `LayoutAnimation` dan animasi indikator header agar transisinya tidak terasa patah.
- Jika sebuah aksi seperti logout langsung memicu redirect dan unmount screen asal, jangan pasang feedback sukses di screen yang akan hilang. Kirim notice sementara lewat store/route state lalu tampilkan alert di screen tujuan yang tetap aktif.
- Jika sebuah setting bernama `Mata Kuliah Dipantau`, jangan langsung pakai untuk menyaring seluruh query UI. Perlakukan itu sebagai scope monitoring/notifikasi kecuali user memang meminta filter tampilan global; kalau tidak, course yang minim data seperti PKL bisa membuat seluruh app tampak kosong.
- Jika feedback sukses sifatnya ringan seperti `Tersimpan`, jangan pakai dialog blocking yang selalu menunggu tap user. Tambahkan auto-dismiss opsional, dan simpan dialog manual untuk info/warning yang memang perlu dibaca lebih lama.
- Jangan tampilkan dua feedback sukses untuk aksi yang sama di screen yang sama. Jika sudah ada popup alert setelah save, hapus banner atau teks sukses kedua di bawah tombol agar respons UI tetap tunggal dan jelas.
- Jika sebuah setting berpotensi ambigu seperti `Mata Kuliah Dipantau`, tambahkan bantuan inline kecil di dekat judul section dan tampilkan penjelasannya on-demand. Ini lebih rapi daripada menambah teks permanen, tapi tetap mencegah salah paham.
- Jangan letakkan tombol bantuan kecil di dalam `Pressable` induk yang juga punya aksi sendiri seperti accordion header. Pisahkan area kliknya sebagai sibling dengan target sentuh yang cukup besar, atau tap bantuan akan sulit dipicu dan UX terasa rusak.
- Jika preload awal menulis data sementara ke cache lalu background hydration menyempurnakannya, jangan biarkan dashboard menghitung KPI dari data mentah itu. Tandai data yang belum final dan tampilkan placeholder ringan sampai versi stabil siap, agar angka tidak terlihat salah lalu berubah sendiri.
- Jika user meminta hero/card branding tertentu tetap dark walau app sedang di mode terang, terapkan palet tetap itu secara lokal pada komponen yang dimaksud. Jangan paksa theme global berubah hanya untuk satu blok visual.
- Jika user sudah mengarahkan gaya branding baru pada dashboard, audit juga card akun atau screen lain yang masih satu konteks visual. Jangan biarkan satu blok brand sudah rapi tetapi blok akun utama masih tertinggal gaya lama.
- Jika user bilang aplikasi sudah mau disebarkan, jangan arahkan ke jalur `preview/internal` atau APK release yang masih signed debug. Langsung arahkan ke toolchain production: JDK, release keystore, dan build signed `AAB`/APK yang layak distribusi.
- Untuk login Moodle di APK/release, jangan kirim kredensial ke `login/token.php` lewat query GET jika tidak perlu. Gunakan `POST` `application/x-www-form-urlencoded` dan `Accept: application/json` agar lebih tahan terhadap redirect, gateway, atau WAF yang membalas HTML.
- Jika integrasi login bergantung pada server kampus yang kadang membalas halaman maintenance HTML, jangan berhenti di error generik non-JSON. Deteksi maintenance secara eksplisit dan tampilkan alert khusus agar user tahu masalahnya ada di server, bukan pada password mereka.
- Jika user memberi referensi visual untuk footer/tab bar dan kartu statistik, ubah lewat komponen shared atau layout navigator terlebih dulu. Jangan tempel style per-screen kalau targetnya adalah pola aktif/nonaktif dan proporsi ikon yang harus konsisten di seluruh app.
- Untuk footer/tab bar yang bergaya floating, state aktif harus tetap subtil. Jangan membesarkan bubble, shell, dan label sekaligus; biasanya cukup satu aksen aktif yang lebih kecil agar navigasi tetap terasa rapi dan tidak menutupi konten.
- Jika referensi menunjukkan active tab berada di dalam footer sebagai pill, jangan pecah lagi jadi bubble ikon terpisah yang naik ke atas. Pertahankan satu container aktif yang utuh agar alignment visualnya stabil.
- Jika footer yang diinginkan benar-benar bergaya dock mengambang, jangan paksa layout lewat `tabBarIcon` bawaan. Pindahkan ke custom `tabBar` penuh supaya shadow, bubble aktif, lebar slot, dan label satu baris bisa dikontrol dengan rapi.
- Jika user meminta footer terasa langsung melayang di atas konten, jangan sisakan panel background besar di belakang semua tab. Pertahankan container layout-nya, tetapi buat surface luarnya transparan dan biarkan aksen aktif yang menjadi fokus visual.
- Jika user memberi anotasi visual yang membedakan wrapper luar dan dock utama, ikuti batas itu secara literal. Hapus blok wrapper yang ditandai, tetapi jangan sekaligus menghapus surface dock utama yang justru ingin dipertahankan.
- Jika user meminta footer “lebih rounded” dan “lebih kecil”, ubah beberapa proporsi sekaligus secara halus: radius dock, tinggi dock, padding, bubble aktif, ikon, dan label. Mengubah satu angka saja biasanya membuat komposisi tetap terasa off.
- Jika dock utama sudah benar dan user hanya ingin footer terasa lebih ringan, kecilkan bubble aktif saja dulu sebelum menyentuh ukuran dock. Ini perubahan paling aman untuk memperhalus visual tanpa menggeser layout keseluruhan.
- Jika user hanya meminta bubble aktif lebih square, jangan sentuh dock utama atau ukuran slot tab. Ubah radius bubble aktif saja agar koreksinya lokal dan tidak membuka regresi alignment baru.
- Jika user meminta lebar tab bar lebih kecil, sempitkan wrapper luar lebih dulu daripada mengecilkan slot tab di dalam dock. Ini menjaga spacing ikon/label tetap stabil sambil mengurangi footprint horizontal footer.
- Jika user mengoreksi bahwa yang harus diperkecil adalah vertikal, jangan terus menyentuh lebar wrapper. Pulihkan ukuran horizontalnya, lalu kecilkan `minHeight`, `paddingVertical`, tinggi bubble aktif, dan line-height label secara terarah.
- Jika user meminta footer/tab terasa lebih square, utamakan penurunan `borderRadius` pada dock dan bubble aktif. Jangan langsung mengubah ukuran atau spacing kalau masalahnya hanya siluet.
- Jika ingin menambah swipe antar-tab pada screen yang banyak `ScrollView`, gunakan wrapper gesture shared di level screen, bukan logika per-halaman acak. Tambahkan exclusion zone untuk area yang memang punya scroll horizontal sendiri seperti filter chips.
- Koreksi 25-Apr-2026: untuk tab utama yang seluruh halamannya sudah dibangun di atas `ScrollView`, jangan pakai swipe hack screen-level berbasis `PanResponder` sebagai mekanisme utama perpindahan tab. Gunakan pager/native swipe milik navigator, lalu tambahkan lock sederhana untuk child horizontal scroll seperti filter chips.
- Koreksi 25-Apr-2026 lanjutan: jika child horizontal scroll seperti filter chips masih membuat gesture pager terasa nyangkut, jangan terus menumpuk lock/unlock gesture. Untuk jumlah opsi yang sedikit, ubah chip menjadi layout wrap non-horizontal agar konflik gesture hilang dari akarnya.
- Setelah mengganti komponen horizontal menjadi layout wrap demi menghilangkan konflik gesture, audit juga hasil visualnya. Bugfix gesture yang benar masih bisa terasa kasar kalau spacing wrapper, tinggi chip, dan bobot label dibiarkan seperti layout lama.
- Untuk filter kategori yang jumlah opsinya 5-6 dan dipakai di mobile, `wrap` bebas sering tetap terasa berantakan. Opsi yang lebih rapi biasanya adalah grid 2 baris dengan lebar item terkontrol, segmented control terbatas, atau ringkas jadi dropdown/sheet jika swipe halaman harus tetap bebas.
- Untuk grid filter mobile yang item-nya tidak habis dibagi kolom, jangan biarkan baris terakhir tersisa kecil di kiri. Buat lebar baris terakhir adaptif (mis. 2 item jadi dua chip lebar yang terpusat) agar layout tetap terlihat intentional.
- Jangan pakai satu jumlah kolom filter untuk semua screen secara buta. `Absensi` dengan 6 opsi cocok 3 kolom, tetapi `Tugas` dengan 4 opsi lebih rapi 2 kolom. Komponen shared perlu dukungan konfigurasi per screen.
- Jika fitur `Riwayat` dibangun di atas source SUNAN yang hanya mengirim sesi upcoming/aktif, jangan asumsikan history lampau akan muncul sendiri. Simpan history lokal per akun untuk item yang pernah terlihat, lalu jelaskan batasannya di empty state agar perilakunya tidak membingungkan user.
- Untuk perbedaan splash antara HP dan emulator Android, audit native resources yang benar-benar dipakai build (`android/res`) dan jangan hanya percaya `app.json`. HP sering menampilkan Android system splash/cold start lebih jelas daripada emulator.
- Jika profile internal memakai package preview terpisah, suffix `Preview` tidak harus ikut ke nama aplikasi yang terlihat user. Pertahankan package terpisah bila perlu, tetapi biarkan label app tetap produksi agar install dan tampilan tidak terasa seperti build sementara.
- Jika user meminta upload seluruh workspace tetapi hanya file yang aman, audit dulu nested repo, `.env`, service-account JSON, dan folder temp lokal. Untuk repo aplikasi yang sudah terpisah, lebih aman referensikan sebagai submodule daripada menyalin isi repo mentah ke repo root.
- Jika user meminta setiap perubahan project di-commit, perlakukan itu sebagai aturan kerja tetap. Setiap perubahan yang saya buat harus ditutup dengan commit yang jelas, bukan dibiarkan hanya sebagai working tree change.
- Jika splash awal di HP tetap jelek walau in-app loading screen sudah benar, jangan terus mengubah React loading screen. Audit Android system splash khususnya `windowSplashScreenAnimatedIcon`, varian `drawable-night`, dan apakah plugin Expo sedang membangkitkan bitmap penuh berlatar belakang yang lalu tampil sebagai kotak di tengah layar.
- Jika filter kategori di screen tab mulai memakan tinggi layar dan terasa ramai, jangan paksa terus tampil inline. Untuk opsi yang tidak perlu selalu terlihat, lebih rapi pindahkan ke tombol floating yang membuka popup filter di atas dock/footer.
- Jika screen memakai dock footer floating dan punya CTA penting di bagian paling bawah seperti tombol logout atau simpan, jangan pakai `paddingBottom` statis pada `ScrollView`. Gunakan spacing dinamis berbasis safe area + tinggi dock agar aksi bawah tidak ketutup footer.
- Jika app memakai dock footer floating di banyak tab, jangan hitung clearance bawah per-screen dengan angka acak. Buat helper shared untuk bottom spacing dan offset FAB supaya jarak konten ke dock konsisten di semua halaman.
- Jika user mengoreksi bahwa hanya sebagian halaman yang terlalu jauh dari dock, jangan ubah semua tab sekaligus lagi. Pertahankan halaman yang sudah pas, lalu pisahkan clearance antara halaman dock-only dan halaman yang punya FAB/filter overlay.
- Jika user masih meminta jarak halaman FAB diperkecil setelah tuning pertama, lakukan penyesuaian kecil bertahap pada helper shared, bukan perubahan besar lagi. Untuk layout bawah, lebih aman turun sedikit-sedikit daripada mengulang reset semua halaman.
- Jika user akhirnya meminta “samakan dengan dashboard”, jangan pertahankan clearance khusus hanya karena ada FAB filter. Samakan baseline content padding-nya dulu dengan halaman referensi, lalu biarkan FAB mengambang di atasnya.
- Jika user bilang popup/filter “masih berantakan”, jangan tambah aksen baru. Kurangi elemen yang redundant dulu: badge ganda, hint text, tile ikon besar, dan divider berlapis biasanya justru membuat popup kecil terasa sesak.
- Jika app memakai theme custom sendiri di atas Expo Router/React Navigation, jangan biarkan `NavThemeProvider` tetap memakai `DefaultTheme` atau `DarkTheme` mentah. Sinkronkan warna navigation theme dan `Stack` `contentStyle` ke palet app agar screen detail tidak memunculkan flash/background terang saat push-pop di mode gelap.
- Untuk integrasi Moodle/SUNAN, jangan klasifikasikan error generik seperti `accessexception` atau HTTP `403` sebagai sesi invalid secara global. Endpoint opsional atau capability-sensitive bisa gagal walau token masih valid; hanya kode yang jelas menunjukkan token/login habis yang boleh memicu `expireSession`.
- Jangan buang sesi hanya berdasarkan auth error dari satu query background. Untuk integrasi yang server-nya sering maintenance atau tidak stabil, verifikasi token sekali lagi lewat endpoint ringan seperti `core_webservice_get_site_info` sebelum benar-benar logout.
- Jika sebuah splash/loading full-screen hidup di dalam route `(tabs)`, jangan anggap header stack akan ikut hilang otomatis. Header tab berasal dari root stack, jadi butuh state boot yang dibaca di level stack agar `headerShown` bisa dimatikan selama gate masih loading.
- Jika flash terang masih muncul saat transisi `back` di Android setelah theme React Navigation sudah benar, audit resource native `AppTheme` juga. `windowBackground`, `statusBarColor`, dan `navigationBarColor` harus mengarah ke color resource yang punya varian `values-night`, bukan hex terang statis di `styles.xml`.
- Jika repo Expo meng-ignore folder generated `android/`, jangan jadikan file di sana sebagai source of truth untuk bugfix native. Pindahkan perubahan ke `app.json`, config plugin, atau plugin lokal Expo yang bisa meregenerasi resource yang sama saat prebuild/build.
- Jika flash putih sangat cepat masih terlihat pada screen detail setelah theme navigation dan native background sudah benar, audit surface screen itu sendiri. Jangan return `ScrollView` langsung; bungkus dengan root `View` ber-background theme, beri `contentContainerStyle` yang juga punya background + `flexGrow`, dan bila perlu pilih animasi route yang lebih aman seperti `fade` daripada slide horizontal.
- Jika screenshot atau rekaman masih menunjukkan stripe tipis screen lama saat push/pop, anggap itu artefak animasi native stack. Jangan terus men-tune background; untuk route yang bermasalah, lebih aman matikan animasi (`animation: 'none'`) daripada mempertahankan transisi yang terlihat rusak.
- Jika hanya jalur kembali dari detail yang masih bermasalah, jangan pakai `router.back()` secara buta. Bawa info asal route saat membuka detail, lalu override back header dan hardware back ke `dismissTo`/route stabil yang spesifik agar tidak memicu pop animation yang rusak.
- Jika sebuah screen detail terbukti aman sebelum penambahan CTA baru, jangan terus menumpuk workaround di detail route. Kembalikan detail screen ke baseline aman, lalu redesign CTA baru agar tetap memakai jalur interaksi lama yang sudah stabil, idealnya satu `Pressable` saja.
- Jika audit git menunjukkan delta functional dari baseline aman tinggal CTA visual kecil, jangan terus menyalahkan CTA itu. Cek dulu apakah fix global navigation theme atau `Stack contentStyle` sempat ter-revert; flash push/pop sering datang dari scene background React Navigation yang fallback ke theme default.
- Untuk screen detail yang ditampilkan lewat stack transition, jangan andalkan `style` pada `ScrollView` sebagai background utama. Root element route harus `View` berwarna penuh; `ScrollView` hanya mengisi kontennya, karena area di belakang layar saat transisi bisa muncul sebelum konten scroll benar-benar dirender.
- Jika theme app disimpan lewat Zustand `persist`, jangan biarkan navigator utama mount sebelum store theme selesai rehydrate. Scene background yang dibaca saat render awal bisa memakai palette default sementara, lalu memunculkan flash terang saat transisi meski theme akhir sebenarnya gelap.
- Jika screen tab menjadi latar belakang untuk route detail, jangan pakai `return <LoadingView />` atau screen penuh terpisah saat query sedang loading ulang. Pertahankan layout dasar tetap mounted dan render loading/error secara inline agar scene di bawah transisi tidak berganti total.
- Jika app punya theme preference sendiri yang bisa berbeda dari mode sistem, jangan anggap resource Android `values-night` otomatis akan mengikuti preference itu. Sinkronkan native app appearance ke theme app saat startup, atau `windowBackground`/scene native masih bisa memilih resource light walau UI React sudah dark.
- Jika sebuah fallback web hanya memperkaya item absensi yang sudah lolos query bulan aktif, jangan asumsi `Riwayat` akan otomatis ikut kaya. Untuk history attendance, enrichment harus dijalankan setelah daftar baru digabung dengan history, dan key merge-nya harus berbasis modul+waktu sesi, bukan `eventId` saja.
- Saat route detail berada di atas navigator tab dalam native stack, cek apakah screen dasar sedang `freezeOnBlur`. Scene yang dibekukan bisa menimbulkan flash atau blank singkat saat back transition, jadi matikan freeze untuk route dasar yang harus tetap siap terlihat di bawah animasi.
- Jika global `screenOptions.contentStyle` belum cukup menghilangkan flash pada native stack, set `contentStyle` langsung pada route yang bermasalah. Opsi route spesifik lebih dekat ke card transition yang sedang dianimasikan dan kadang lebih deterministik daripada hanya mengandalkan inheritance global.
- Jika user akhirnya lebih mementingkan UX stabil daripada transisi screen penuh, jangan terus memaksa route biasa. Ubah detail menjadi popup/modal transparan agar entry/exit tidak lagi bergantung pada back transition native stack yang bermasalah.
- Jika header tab membaca pathname global untuk menentukan judul, route overlay seperti `/task/:id` bisa membuat resolver jatuh ke fallback default. Simpan tab terakhir yang valid dan pakai itu selama modal/overlay terbuka agar header belakang tetap konsisten.
- Untuk integrasi Moodle, jangan klasifikasikan semua `accessexception` atau `403` sebagai sesi habis. Bedakan token benar-benar invalid dari access/permission/service errors, terutama pada endpoint opsional seperti quiz, kalender, atau submission status, atau app akan logout sendiri walau masalahnya bukan pada sesi user.
- Jangan andalkan `addNotificationResponseReceivedListener` saja untuk deep-link notifikasi. Saat app dibuka dari keadaan mati, baca juga `getLastNotificationResponseAsync()` saat bootstrap agar tap notifikasi tidak hilang sebelum listener sempat subscribe.
- Jika query screen bergantung pada query upstream seperti `courses`, jangan biarkan screen hanya membaca query turunan yang bisa `enabled: false`. Prioritaskan loading/error upstream dulu, atau user akan melihat empty state palsu padahal masalah aslinya ada pada fetch parent.
- Untuk event absensi tanpa `timeduration`, jangan biarkan status `available` selamanya setelah `timestart` lewat. Minimal turunkan ke `closed` setelah hari lokal berganti agar filter `Riwayat` tidak kehilangan sesi lampau.
- Untuk notifikasi yang diklaim “sekali per hari” atau “sekali per window”, jangan dedupe hanya dengan `useRef(new Set())` di komponen. Force-close lalu buka lagi akan mengirim ulang. Simpan dedupe key di storage persisten dan prune key lama secara berkala.
- Jika sebuah setting sudah disimpan ke backend untuk lintas device/reinstall, jangan berhenti di jalur save saja. Tutup loop-nya: schema harus punya kolomnya, payload save harus mengirimnya, dan app harus memuatnya lagi saat sesi authenticated aktif.
- Untuk sync settings ke backend, jangan biarkan satu kolom baru seperti `notify_task_open` menjatuhkan seluruh upsert/select. Jalur save/load harus kompatibel dengan schema lama, dan `appUserId` perlu bisa dipulihkan on-demand tanpa memaksa user login ulang.
- Untuk pesan yang dilihat user, jangan bocorkan detail teknis seperti `server`, `sinkronisasi`, `schema`, atau status parsial jika itu tidak membantu tindakan user. Utamakan hasil yang sederhana dan langsung dimengerti.
- Saat mengaudit copy user-facing, prioritaskan sumber pesan global seperti error helper, dialog update, dan alert login lebih dulu. Satu perubahan di titik itu membersihkan banyak layar sekaligus tanpa perlu edit satu per satu.
- Untuk audit mikrocopy, lanjutkan dari helper global ke komponen shared loading/empty state, baru ke screen yang paling sering dilihat. Pola ini membuat nada bahasa cepat seragam tanpa menyapu seluruh codebase secara buta.
- Untuk notifikasi dan dialog update, pakai kalimat yang lebih pendek daripada copy layar biasa. User membacanya cepat, jadi judul dan isi harus langsung ke inti tanpa istilah campuran seperti `task/quiz`, `APK`, atau instruksi berlapis.
- Saat audit copy user-facing, jangan berhenti di alert dan empty state. Cek juga label status, filter, legend, dan modal lama; istilah seperti `submit`, `quiz`, `pending`, atau daftar sprint internal mudah tertinggal dan tetap terasa teknis bagi user awam.
- Jika sebuah fitur update bergantung pada artefak eksternal seperti manifest JSON dan file APK publik, jangan berhenti di checker runtime. Selalu siapkan template manifest, dokumentasi hosting, dan warning env yang eksplisit agar user tidak salah mengisi URL EAS Update ke jalur APK manual.
- Jika build lokal Android terlihat “hang” hanya dari timeout wrapper atau proses background, jangan langsung simpulkan Gradle macet. Jalankan ulang dengan `--console=plain --no-daemon` dan log yang bisa dibaca dulu; sering kali build sebenarnya sehat dan masalahnya hanya ada di harness eksekusi.
- Jika Expo Android memakai `edgeToEdgeEnabled`, jangan biarkan native-stack header mengandalkan default `statusBarTranslucent`. Paksa `statusBarTranslucent: true` di root stack agar top inset header tidak salah dan judul tab tidak naik ke area jam/sinyal HP.
- Jika header native-stack tetap overlap di device nyata setelah tuning inset, jangan terus memaksa opsi header bawaan. Matikan header `(tabs)` dan pindahkan ke header custom in-screen berbasis `useSafeAreaInsets`, karena itu lebih deterministik lintas Expo Go dan APK.
- Jika Expo Go meledak dari wrapper dev internal seperti `Unable to activate keep awake`, jangan buru-buru menyalahkan screen app. Audit dependensi dev yang di-import otomatis oleh Expo, lalu shim modul itu di Metro bila fitur tersebut memang tidak dipakai app.
- Jika perlu membungkus `expo start` di Windows dengan script Node, jangan spawn `npx.cmd` dari wrapper itu. Jalankan local Expo CLI langsung lewat `node <expo-cli>`, karena spawn `npx` dari Node wrapper bisa gagal atau lebih rapuh di PowerShell/Job Windows.
- Jika wrapper `expo start` perlu tetap menampilkan QR, jangan pipe `stdout` Expo ke process Node sendiri. Biarkan `stdout` tetap `inherit` agar Expo melihat TTY interaktif; tangkap hanya `stderr` bila memang perlu mendeteksi kegagalan tunnel.
- Jika app didistribusikan sebagai APK lokal dan memakai EAS Update, jangan anggap publish EAS otomatis hidup hanya karena `app.json` punya `updates.url` dan `runtimeVersion`. Sebelum menyatakan prompt update akan muncul, cek manifest Android hasil prebuild/build dan pastikan `expo.modules.updates.ENABLED=true`, runtime version benar, dan `UPDATES_CONFIGURATION_REQUEST_HEADERS_KEY` sudah menanam channel yang tepat.
- Untuk EAS Update, jangan campuradukkan `branch` dan `channel`. Publish ke branch saja belum cukup; binary standalone meminta update lewat `expo-channel-name`, jadi selalu verifikasi channel itu benar-benar ada dan menunjuk ke branch yang berisi update terbaru.
- Jika sebuah prompt update bisa ditunda dengan aksi seperti `Nanti`, selalu sediakan titik masuk manual di dalam app untuk membuka lagi atau melanjutkan update itu. Jangan paksa user menunggu startup check berikutnya tanpa jalur eksplisit di Pengaturan atau layar terkait.
- Jika user berkata fitur baru “belum ada” padahal source sudah benar, jangan berhenti di git push. Verifikasi juga jalur distribusinya yang dipakai user (EAS update, APK, branch/channel) sebelum menyatakan perubahan itu sudah bisa dilihat di perangkat.
- Untuk EAS Update yang boleh ditunda dengan `Nanti`, jangan andalkan `checkForUpdateAsync()` saja. Update yang sudah terunduh tetapi belum diterapkan perlu dibaca juga dari state native `expo-updates` seperti `isUpdatePending`, atau user bisa kehilangan tombol untuk melanjutkan update.
- Saat mengetes prompt EAS bersama user, selalu cek dulu apakah perangkat mereka kemungkinan sudah berada di update terbaru. Jika iya, prompt memang tidak akan muncul lagi sampai ada group yang lebih baru; jangan menyuruh user membuka-tutup app berulang tanpa memastikan ada update yang benar-benar lebih baru.
- Jika update sudah tersedia tetapi ditunda, jangan biarkan tombol sekunder di Pengaturan tetap berarti “cek ke server”. Itu membingungkan karena server bisa menjawab “sudah terbaru” padahal update yang ditunda masih siap dipakai. Untuk kondisi deferred, tombol sekunder harus membuka lagi update yang sudah ada.
- Jika popup update sudah menjadi CTA utama, jangan duplikasi aksi `Update sekarang` lagi di kartu Pengaturan. Pengaturan cukup menjadi titik `cek` atau `lihat update`, sedangkan aksi pasang/update tetap lewat satu dialog yang konsisten.
- Untuk log sesudah update, jangan mengandalkan `targetUpdateId` dari runtime update yang baru diunduh saja. Simpan juga identitas versi/update yang sedang berjalan sebelum update dimulai, lalu tampilkan log ketika app terbukti sudah berpindah ke versi atau update yang berbeda.
- Jika effect React membaca state update native seperti `isUpdatePending` lalu menulis object baru ke store global, selalu beri guard equality dulu. Menulis object EAS yang identik pada setiap render akan memicu loop `maximum update depth exceeded`.
- Jika sebuah effect update masih rawan rerender karena dependency dari `useUpdates()` sering berganti referensi, jangan hanya bandingkan isi object sekali. Tambahkan ref-based idempotency key agar write ke store benar-benar terjadi satu kali per event logis.
- Jika coordinator update menulis payload yang sama ke Zustand lebih dari satu tempat, jangan hanya menambah guard di effect caller. Tambahkan no-op equality guard di action store juga, supaya write identik tidak pernah memicu rerender loop walau ada caller lain yang masih terlalu rajin menulis state yang sama.
- Jika popup update boleh ditutup dengan tap di luar card, jangan simpan ketersediaan update hanya di state memori sementara. Persist satu payload “update ditunda” dan jadikan itu sumber reopen di Pengaturan, atau user bisa melihat prompt lalu beberapa detik kemudian justru mendapat pesan “sudah terbaru”.
